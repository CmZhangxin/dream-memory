## 实施过程发现的问题与修复

### 问题 1：vitest fakeTimers 无法拦截 AbortController 内部 setTimeout

**现象**：Task 2 的 timeout 测试用 `vi.useFakeTimers()` + `advanceTimersByTime(11_000)`，实际等了 5s 后 vitest 默认 testTimeout 触发失败。

**根因**：`AbortController.abort()` 由 `setTimeout` 触发，但 vitest 的 fake timers 不替换模块加载时绑定的 setTimeout 引用（特别是构造函数里 `this.setTimeout = setTimeout` 这种）。

**修复**：改为真实超时方案 —— `new GatewayClient({ timeoutMs: 50 })`，mock fetch 时只在 `signal.addEventListener('abort')` 上 reject。58ms 测试通过。

### 问题 2：MCP SDK `registerTool` `inputSchema` 类型不匹配

**现象**：Task 7 typecheck 报 `Type 'Record<string, unknown>' is not assignable to type 'AnySchema | ZodRawShapeCompat | undefined'`。计划文档让我传 `(tool.inputSchema as { shape: ... }).shape`。

**根因**：MCP SDK 1.29 的 `registerTool` 重载有两组：`ZodRawShape`（zod object 的 `.shape`）和 `AnySchema`（整个 zod schema）。把 `.shape` 强转 `Record<string, unknown>` 时 TS 反推不到 ZodRawShape。

**修复**：直接传整个 zod schema（走 AnySchema 重载），SDK 内部能处理。代码更简洁，类型检查通过。

### 问题 3（已记录，未修复）：StreamableHTTPServerTransport 多客户端可能共享 server 实例

**现象**：当前实现里每个新 sessionId 创建一个新 transport，但都连到同一个 `McpServer` 实例。理论上 SDK 应支持，但 spec 说"per-session transport"是否真隔离 state 没实测。

**影响**：当前架构 `dream_*` 工具是无状态的（除了 session_key），所以即便 server 共享也不影响功能。

**留给阶段 4 处理**：dashboard 接入 SSE 时如果出现并发 session 串扰，再调整为 per-session McpServer。
