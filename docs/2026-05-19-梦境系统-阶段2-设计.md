## 一、目标与范围

### 1.1 一句话目标

让 CodeBuddy / Claude Desktop / 任何 MCP 兼容客户端，**通过统一的 `dream_*` 工具集**调用 `~/.dream-memory/` 中的 Hermes Memory Gateway，实现"记忆写入 / 检索 / 升华"全流程。

### 1.2 范围（务实版 B · 8 工具）

**直接对应上游 7 个 endpoint 的 5 个工具**：

| MCP 工具 | 上游 endpoint | 用途 |
|---|---|---|
| `dream_capture_conversation` | `POST /capture` | 写入 user/assistant 一来一回 |
| `dream_recall` | `POST /recall` | 基于 query 拉相关记忆作 system context |
| `dream_search_memories` | `POST /search/memories` | 搜 L1 atoms |
| `dream_search_conversations` | `POST /search/conversations` | 搜 L0 原始对话 |
| `dream_session_end` | `POST /session/end` | 结束会话 + 触发 flush 升华 |

**包装上游 `/capture` 的 3 个语义化变体**：

| MCP 工具 | 实现策略 |
|---|---|
| `dream_capture_note` | 把 `content` 包成 `{user:"记一笔: <content>", assistant:"已记录"}` 调 `/capture` |
| `dream_capture_decision` | 把 `{what, why, alternatives?}` 拼成结构化文本，包成对话 |
| `dream_capture_skill` | 把 `{name, when, how, examples?}` 拼成 `#skill` 标记的对话 |

**Stub 工具（抛 `NotImplementedError`，留接口位置）**：

| MCP 工具 | 状态 | 备注 |
|---|---|---|
| `dream_get_persona` | 🚧 Stub | 上游无 L3 Persona API |
| `dream_get_today` | 🚧 Stub | 上游无按日期查询 |
| `dream_get_scenarios` | 🚧 Stub | 上游无 L2 列表 API |
| `dream_capture_task` | 🚧 Stub | 等 todo 域语义明确再做 |

**总计**：8 个真实工具 + 4 个 Stub = 12 个工具（与设计文档列表对齐）。

### 1.3 不在阶段 2 范围

- ❌ 同步脚本（`sync-vault.ts` / `sync-codebuddy.ts`）→ 阶段 3
- ❌ Skills 编排（capture-decision / daily-review / dream-now）→ 阶段 3
- ❌ Dashboard 改造 → 阶段 4
- ❌ npm 发布

## 二、技术栈与协议

### 2.1 技术栈

| 项 | 选型 | 备注 |
|---|---|---|
| 语言 | TypeScript 5（strict + ESM） | |
| 运行时 | Node.js ≥ 22.16 | 与阶段 1 一致 |
| MCP SDK | `@modelcontextprotocol/sdk` ^1.29.0 | 官方 TS SDK |
| 校验 | `zod` ^3.x | MCP 工具输入 schema |
| HTTP 客户端 | Node 内置 `fetch`（Node 22 原生） | 不引入 axios/undici |
| 测试 | `vitest` ^1.x | 含 mock Gateway |
| 包管理 | npm（与项目其他子目录一致） | |

### 2.2 协议支持（你选了 (c) 全协议）

| Transport | SDK 内置 | 用途 |
|---|---|---|
| **stdio** | ✅ | CodeBuddy / Claude Desktop 本地 |
| **HTTP/SSE** | ✅ | 浏览器 / 远程 Agent；MCP SDK ≥1.0 已用 "Streamable HTTP" 取代旧 SSE，本项目同时支持新旧两种以兼容 |
| **WebSocket** | ❌ 需自实现 transport | 长连接场景；按 MCP 协议规范写一个 `WebSocketServerTransport` |

启动方式：
```bash
dream-memory-mcp                       # 默认 stdio（被 MCP 客户端 spawn）
dream-memory-mcp --transport http      # HTTP/SSE，监听 :8421
dream-memory-mcp --transport ws        # WebSocket，监听 :8422
dream-memory-mcp --transport all       # 三种同时
```

> 端口选择避开 8420（Hermes Gateway 占用）。

### 3.2 关键设计原则

| 原则 | 落地 |
|---|---|
| **单一职责** | 每个工具一个文件，文件 < 100 行 |
| **依赖倒置** | 工具不直接 fetch，统一走 `GatewayClient` |
| **可测试** | 工具函数纯：`(input, ctx) => result`；ctx 包含 mock 友好的 client |
| **错误透明** | Gateway 错误 → MCP 错误响应（含 isError: true 和具体原因），不吞 |
| **零状态** | 每次工具调用独立；session_key 由调用方传或自动生成 |

## 四、核心组件设计

### 4.1 GatewayClient

```typescript
// gateway-client.ts
export class GatewayClient {
  constructor(private baseUrl: string = "http://localhost:8420") {}

  async health(): Promise<HealthResponse> { ... }
  async recall(req: RecallRequest): Promise<RecallResponse> { ... }
  async capture(req: CaptureRequest): Promise<CaptureResponse> { ... }
  async searchMemories(req: MemorySearchRequest): Promise<MemorySearchResponse> { ... }
  async searchConversations(req: ConversationSearchRequest): Promise<ConversationSearchResponse> { ... }
  async sessionEnd(req: SessionEndRequest): Promise<SessionEndResponse> { ... }
  async seed(req: SeedRequest): Promise<SeedResponse> { ... }
}
```

**关键决策**：

- 类型定义**直接复制**自 `~/.dream-memory/tdai-memory-openclaw-plugin/src/gateway/types.ts`（防上游变更未察觉，本地版本是契约快照）
- 超时：默认 10s（与设计文档第 9.2 节"MCP 工具响应 10 秒 timeout"对齐）；`/seed` 超时 5min
- 重试：仅对网络错误重试 1 次；HTTP 4xx/5xx 直接报错
- **不**吞 Gateway 错误：直接传给 MCP 客户端（client 决定如何降级）

### 4.2 Session Management

```typescript
// session.ts
// session_key 生成策略：
//   - 显式传入 → 用之
//   - 否则 → "dream-mcp-{YYYYMMDD}-{client_hint}"
//   - client_hint 从 process.env.MCP_CLIENT_NAME 或 stdio 启动参数读
//   - 兜底：当日纪元秒
```

`session_key` 是上游强 schema 字段，不能为空。设计文档第 5.1 节没有具体规定生成策略，我们定如下：

| 来源 | session_key 形态 | 例 |
|---|---|---|
| 工具显式传 `session_key` | 原样透传 | `proj-workshop-2026-05-19` |
| 未传 + CodeBuddy stdio | `dream-mcp-codebuddy-YYYYMMDD` | `dream-mcp-codebuddy-20260519` |
| 未传 + Claude stdio | `dream-mcp-claude-YYYYMMDD` | |
| 未传 + 远程 HTTP | `dream-mcp-remote-{ip-hash}-YYYYMMDD` | |

### 4.3 工具实现模板

```typescript
// tools/capture-note.ts
export const captureNoteSchema = z.object({
  content: z.string().min(1).describe("要记录的笔记正文"),
  tags: z.array(z.string()).optional().describe("标签（如 #skill #decision）"),
  session_key: z.string().optional().describe("会话标识，未传则自动生成"),
});

export async function captureNote(
  input: z.infer<typeof captureNoteSchema>,
  ctx: ToolContext,
) {
  const tags = input.tags?.map((t) => t.startsWith("#") ? t : `#${t}`).join(" ") ?? "";
  const userText = `[NOTE] ${input.content}${tags ? "\n" + tags : ""}`;

  const result = await ctx.client.capture({
    user_content: userText,
    assistant_content: "已记录",
    session_key: input.session_key ?? ctx.session.current(),
  });

  return {
    content: [{
      type: "text" as const,
      text: `✅ 笔记已写入 L0：${result.l0_recorded} 条记录`,
    }],
  };
}
```

## 六、错误处理

| 失败场景 | MCP 工具行为 |
|---|---|
| Gateway 未启动（ECONNREFUSED） | 返回 `isError: true` + "Memory Gateway 未启动，请运行 `cd dream-dashboard && ./start.sh`" |
| Gateway 返回 4xx | `isError: true` + 透传 error message |
| Gateway 返回 5xx | `isError: true` + "Gateway 内部错误，详见 ~/.dream-memory/gateway.log" |
| 工具入参 zod 校验失败 | MCP SDK 自动返回 InvalidParams（-32602） |
| 工具调用超时（10s） | `isError: true` + "Gateway 响应超时" |
| Stub 工具被调用 | `isError: true` + "暂未实现，将在阶段 X 提供" |

## 九、风险与 YAGNI

| 风险 | 等级 | 缓解 |
|---|---|---|
| 上游 Gateway HTTP 协议变更 | 中 | 类型定义本地复制，CI 加 schema diff 检查（阶段 2 先不做，记 backlog）|
| WebSocket transport 自实现 bug | 中 | 集成测试覆盖 + 限定先 stdio 用得起来再切 |

| 4 个 stub 工具迷惑用户 | 低 | tool description 明确写 "Not yet implemented in Stage 2" |
| session_key 自动生成造成 L1 升华跨日错乱 | 中 | 默认按日切；用户显式传时尊重 |
