让 Obsidian Vault 和 CodeBuddy 项目 memory 日报**手动一行命令同步**进 Hermes Gateway，跑完能在 `dream_search_conversations` 里检索到。

## 四、数据流设计

```
┌──────────────────────────────────────────────────────────────┐
│  数据源                                                        │
│                                                              │
│  ① Obsidian Vault                                            │
│     ~/Documents/Obsidian Vault/**/*.md  (64 个 md)          │
│                                                              │
│  ② CodeBuddy Memory（白名单项目）                              │
│     <project>/.codebuddy/memory/YYYY-MM-DD.md               │
└──────┬───────────────────────────────────────────────────────┘
       │
       ▼ 1. 读取 + 增量过滤（mtime > sync-state）
┌──────────────────────────────────────────────────────────────┐
│  处理 pipeline                                                 │
│                                                              │
│  redact()  → 脱敏（密码/token/邮箱/手机号）                     │
│  chunk()   → vault 按 \n\n 切段；codebuddy memory 整篇       │
│  enrich()  → 加来源标记 [VAULT:path] / [CB-MEMORY:date]      │
└──────┬───────────────────────────────────────────────────────┘
       │
       ▼ 2. POST /capture（一段=一次调用）
┌──────────────────────────────────────────────────────────────┐
│  Hermes Gateway :8420                                         │
│  L0 写入 → 异步触发升华                                        │
└──────┬───────────────────────────────────────────────────────┘
       │
       ▼ 3. 全部完成后 POST /session/end
┌──────────────────────────────────────────────────────────────┐
│  Hermes Gateway 升华 pipeline 启动 L0→L1→L2                    │
└──────────────────────────────────────────────────────────────┘
       │
       ▼ 4. 写回 sync-state.json
       记录每个文件的 mtime hash，下次跳过
```

## 五、关键设计决策

### 5.1 Session key 命名

每次跑 `daily-dream` 用一个**统一的 session_key**，让升华阶段把同批同步的内容当成一次"会话"：

```
sync-{source}-YYYYMMDD
例：
  sync-vault-20260519
  sync-codebuddy-20260519
```

> 不和 dream-mcp 的 `dream-mcp-{client}-YYYYMMDD` 冲突 —— 前缀不同。

### 5.2 capture 请求格式

每条 capture 包成"伪对话"：

```ts
POST /capture
{
  "user_content": "[VAULT:00-收件箱/想法.md] §3\n\n这一段的实际内容",
  "assistant_content": "已收录",
  "session_key": "sync-vault-20260519"
}
```

`[VAULT:...]` 前缀是**来源标记** —— 后续 search 时可按前缀过滤。

### 5.3 增量状态

`~/.dream-memory/sync-state.json`：

```json
{
  "schema_version": 1,
  "vault": {
    "last_sync_at": "2026-05-19T12:00:00Z",
    "files": {
      "00-收件箱/想法.md": {
        "mtime_ms": 1747652400000,
        "size_bytes": 2345,
        "chunks_synced": 8
      }
    }
  },
  "codebuddy_memory": {
    "last_sync_at": "2026-05-19T12:00:00Z",
    "files": {
      "/Users/cmzhang/worker/job/workshop/.codebuddy/memory/2026-03-17.md": {
        "mtime_ms": 1714440000000,
        "size_bytes": 13191
      }
    }
  }
}
```

判定"已同步"：`current.mtime_ms === stored.mtime_ms` 跳过。

### 5.4 段落切分（仅 Vault）

```ts
function chunk(content: string): string[] {
  // 1. gray-matter 剥离 frontmatter（不入 L0）
  // 2. 按 \n\n+ 切段
  // 3. 过滤空段 + 长度 < 30 字符的废段
  // 4. 单段 > 2000 字符的截短到 2000（保留前缀提示）
}
```

CodeBuddy memory 不切分（已经是结构化日报，整篇当一条更利于 L1 升华看上下文）。

### 5.5 敏感词过滤

```ts
const PATTERNS: Array<{name: string, re: RegExp, replacement: string}> = [
  { name: "openai-key",   re: /sk-[A-Za-z0-9_-]{20,}/g,                 replacement: "[REDACTED:KEY]" },
  { name: "github-token", re: /gh[pousr]_[A-Za-z0-9]{36,}/g,            replacement: "[REDACTED:TOKEN]" },
  { name: "email",        re: /\b[\w.+-]+@[\w-]+\.[\w.-]+\b/g,          replacement: "[REDACTED:EMAIL]" },
  { name: "phone-cn",     re: /(?<!\d)1[3-9]\d{9}(?!\d)/g,              replacement: "[REDACTED:PHONE]" },
  { name: "password-kv",  re: /(?:password|passwd|pwd)\s*[:=]\s*\S+/gi, replacement: "[REDACTED:PASSWORD]" },
];
```

策略：**命中则替换为占位符，不丢弃整段**。日志计数命中次数（每次同步报告"脱敏 X 次"）。

### 5.6 错误处理

| 场景 | 行为 |
|---|---|
| Gateway 返回 4xx/5xx | 单条记日志，跳过该 chunk，**不**中断整体同步 |
| Gateway 不可达（ECONNREFUSED）| 整个同步立刻中止；返回非 0 退出码 |
| 文件读取失败 | 跳过该文件 + 日志 |
| sync-state.json 损坏 | 警告 + 视为空 state（全量重跑）|
| 敏感词过滤命中 | 替换 + 计数，**不**跳过 |

### 5.7 不引入 dream-memory-mcp 依赖

理由：
- 脚本只用 `/capture` + `/session/end` 两个 endpoint，没必要拽进 MCP SDK + ws 这一坨
- 脚本和 mcp 项目类型契约**独立维护**（都从上游 types.ts 抄）—— 上游变更时各自感知

## 六、CLI 接口

```bash
# 单独同步 Vault
cd workshop/scripts
npx tsx src/sync-vault.ts [--dry-run]

# 单独同步 CodeBuddy memory
npx tsx src/sync-codebuddy.ts [--dry-run]

# 全流程（推荐）
npx tsx src/daily-dream.ts [--dry-run] [--no-flush]
```

参数：
- `--dry-run`：只扫描不写 Gateway，输出"本次会同步 N 段"
- `--no-flush`：跳过末尾的 `/session/end`（调试用）

退出码：
- `0`：成功（即使 0 文件需要同步也是 0）
- `1`：Gateway 不可达
- `2`：配置文件错误
- `3`：未预期错误

## 九、风险

| 风险 | 等级 | 缓解 |
|---|---|---|
| 64 个 md 段落切完 → 数百次 capture，把 LLM Token 烧光 | 中 | 增量 + 段落数量限制（单文件最多 50 段，超的截断）|
| 敏感词正则误报（把正常 `1[3-9]xxx` 数字串当手机号）| 低 | 仅替换不丢弃，命中率会被报告，可调 |
| sync-state.json 并发写入冲突 | 低 | 单进程使用，3b 加 cron 时也是串行；不加锁 |
| Vault 内 binary（图片/PDF）.md 是文本，但内容可能是 base64 | 低 | chunk 单段 > 2000 截断 + 粗暴 |
| 上游 Hermes `/capture` 返回成功但 LLM 异步升华失败 | 低 | 不在脚本范围；脚本只看 HTTP 200 即认为同步完成 |
