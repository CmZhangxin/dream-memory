## 1. 目标与边界

### 1.1 目标

- 让 `daily-dream` 每晚 23:00 自动执行
- 跟随 `dream-dashboard` 启停（关闭 dashboard 即停 cron）
- 失败不阻塞，日志可追溯

### 1.2 不做（YAGNI）

- ❌ 系统级 launchd / crontab（不需要"用户登出后还跑"）
- ❌ 失败重试机制（同步幂等，第二天再跑就行）
- ❌ 多时间点（23:00 + 03:00 兜底等）
- ❌ 失败系统通知（osascript）
- ❌ 启动时立即跑一次（开发期用 `npm run sync` 即可）

### 1.3 设计决策（已确认）

| # | 决策 | 选择 |
|---|------|------|
| 1 | 调度方式 | Node 进程常驻 + `node-cron` |
| 2 | 执行时间 | 每晚 23:00（cron `0 23 * * *`）|
| 3 | 失败处理 | 写日志 + 等下次定时；不重试 |
| 4 | 启动是否跑一次 | 否（只等 cron） |
| 5 | start.sh 集成 | `launch_cron` 函数 + `CRON_ENABLED` 开关，缺 LLM Key 时降级跳过 |

## 2. 架构

### 2.1 进程模型

```
start.sh
├── launch_obsidian()
├── prepare_memory() + launch_memory()  → port 8420
├── launch_cron()  ← 阶段 3b 新增
│   └── node scripts/dist/cron-runner.js  (常驻)
│       └── node-cron 注册 "0 23 * * *" → spawn npx tsx daily-dream.ts
└── launch_dashboard()  → port 3001
```

### 2.2 cron-runner 职责

1. 启动时校验 cron 表达式合法
2. 用 `node-cron` 注册任务
3. 时间到 → spawn `npx tsx src/daily-dream.ts`
4. 子进程 stdout/stderr 写入 `~/.dream-memory/cron.log`（rotate by date）
5. 子进程退出码非 0 → 日志记 ERROR，但 runner 不退出
6. SIGTERM/SIGINT → 优雅关闭

### 2.3 单实例锁（防止重复启动）

PID 文件 `~/.dream-memory/cron-runner.pid`：
- 启动时检查：若文件存在且 PID 还在跑 → 报错退出
- 启动成功后写入自己 PID
- 退出时删除

### 2.4 日志策略

文件：`~/.dream-memory/cron.log`

格式（按行 JSON，便于日后接入 dashboard）：

```jsonl
{"ts":"2026-05-19T20:55:00.000Z","level":"info","event":"cron-runner.started","schedule":"0 23 * * *"}
{"ts":"2026-05-19T23:00:00.000Z","level":"info","event":"sync.start"}
{"ts":"2026-05-19T23:00:32.000Z","level":"info","event":"sync.done","exitCode":0,"durationMs":32000}
{"ts":"2026-05-20T23:00:31.000Z","level":"error","event":"sync.failed","exitCode":1,"durationMs":31000,"stderr":"..."}
```

不做日志切片（每天就 3-5 条，一年 1500 条不到 200KB）。

### Task 2：cron-runner.ts 实现 + TDD（60min）

**Files**:

**测试用例（4 个）**：

| # | 测试 | 期望 |
|---|---|---|
| 1 | 非法 cron 表达式 → 抛错 | `validateSchedule("invalid")` throws |
| 2 | 合法 cron 表达式 → 通过 | `validateSchedule("0 23 * * *")` returns true |
| 3 | PID 文件存在但进程已死 → 可启动 | `acquireLock()` 成功 |

**实现要点**：
- `validateSchedule(expr)` 用 `cron.validate()` 包装
- `acquireLock(pidFile)` 检查 PID 文件 + `process.kill(pid, 0)` 探测进程
- `releaseLock(pidFile)` 删 PID 文件
- `runDaily()` spawn `npx tsx src/daily-dream.ts`，捕获 stdout/stderr 写日志
- `main()` 装好 lock + cron + SIGINT/SIGTERM handler，然后 `process.stdin.resume()` 常驻

### Task 3：start.sh 集成（20min）

**Files**:

- Modify: `dream-dashboard/.env.local.example`（加 `CRON_ENABLED=true` 注释）

**新增逻辑**：

```bash
# .env 默认值
: "${CRON_ENABLED:=true}"
CRON_LOG="$HOME/.dream-memory/cron.log"
CRON_PID_FILE="$HOME/.dream-memory/cron-runner.pid"
CRON_RUNNER="$WORKSHOP_ROOT/scripts/src/cron-runner.ts"  # 用 tsx 跑

launch_cron() {
  if [ "$CRON_ENABLED" != "true" ]; then
    echo "ℹ️  CRON_ENABLED=false，跳过 cron-runner"
    return 0
  fi
  # 缺 LLM Key 时同步无意义（Memory Gateway 没启动）
  if [ -z "${MEMORY_TENCENTDB_LLM_API_KEY:-}" ]; then
    echo "⚠️  缺少 MEMORY_TENCENTDB_LLM_API_KEY，跳过 cron-runner（同步无意义）"
    return 0
  fi
  echo "⏰ 启动 cron-runner（每晚 23:00 自动同步）..."
  cd "$WORKSHOP_ROOT/scripts"
  nohup npx tsx src/cron-runner.ts >> "$CRON_LOG" 2>&1 &
  echo $! > "$CRON_PID_FILE"
  cd - > /dev/null
}
```

需要确定 `WORKSHOP_ROOT` 的来源 —— 实施时先 grep `start.sh` 看现有路径变量怎么命名的，沿用那一套。

**新增逻辑**：

```bash
CRON_PID_FILE="$HOME/.dream-memory/cron-runner.pid"

stop_cron() {
  if [ -f "$CRON_PID_FILE" ]; then
    local pid
    pid=$(cat "$CRON_PID_FILE" 2>/dev/null)
    if [ -n "$pid" ] && kill -0 "$pid" 2>/dev/null; then
      echo "⏹  停止 cron-runner (PID $pid)"
      kill -TERM "$pid" 2>/dev/null
    fi
    rm -f "$CRON_PID_FILE"
  fi
}
```

## 4. 风险与降级

| 风险 | 影响 | 缓解 |
|---|---|---|
| node-cron 时区问题 | 凌晨跑成中午 | 显式传 `{ timezone: "Asia/Shanghai" }` |
| 23:00 时电脑睡眠 | cron 错过 | 当前 macOS 默认行为；接受（YAGNI），未来加 launchd 解决 |
| sync 跑了 30s 仍未结束就被打断（如关 dashboard）| 数据已部分入库 OK，sync-state 可能没 flush | 同步是幂等增量，下次再跑会跳过已成功 + 重做未完 |
| PID 文件被遗弃（runner 崩溃） | 下次启动报"already running" | `acquireLock` 探测进程实际存在 |
