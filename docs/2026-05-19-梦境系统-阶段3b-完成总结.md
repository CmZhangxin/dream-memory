## 3. cron-runner 架构

```
        start.sh
           │
           ├── launch_memory() → :8420
           ├── launch_cron() ─────────┐
           │     │                     │
           │     │ nohup npx tsx       │
           │     │   src/cron-runner.ts│
           │     │   >> cron.log 2>&1  │
           │     │                     ▼
           │                  ┌────────────────────┐
           │                  │  cron-runner       │
           │                  │  ─ acquireLock     │
           │                  │  ─ validateSchedule│
           │                  │  ─ task.schedule(  │
           │                  │      "0 23 * * *", │
           │                  │      runDaily,     │
           │                  │      tz: Asia/SH)  │
           │                  │  ─ SIGTERM/SIGINT  │
           │                  │    handler         │
           │                  └─────┬──────────────┘
           │                        │
           │ 23:00 触发              │ spawn child
           │                        ▼
           │                  npx tsx daily-dream.ts
           │                  ↓ stdout / stderr
           │                  cron.log
           │                  
           └── launch_dashboard() → :3001

        stop.sh
           ├── stop_cron()         ← SIGTERM cron-runner（PID 文件）
           ├── stop_by_port 3001   ← Dashboard
           ├── stop_by_port 8420   ← Memory Gateway
           └── stop_residue()      ← cron-runner 兜底（pgrep）
```

## 4. 关键设计点

### 4.1 单实例锁（PID 文件）

`~/.dream-memory/cron-runner.pid` 保存进程 PID。`acquireLock`：
- 文件存在且 PID 真实活着 → 抛错 "already running"
- 文件存在但 PID 已死（stale）→ 删除并继续
- 文件不存在 → 直接创建

### 4.2 失败语义

| 失败点 | 处理 |
|---|---|
| 启动时 `acquireLock` 抛错 | `main()` 优雅 return（不 exit），保护已运行的 task |
| 子进程 `daily-dream` 失败 | log `sync.failed` + exitCode + stderr 截断 1KB；runner 继续存活 |
| 日志写入失败（磁盘满等） | silent catch，不 crash |

### 4.3 时区

显式设 `Asia/Shanghai`，用 `node-cron` `{ timezone }` 选项。

### 4.4 日志格式（JSON Lines）

```jsonl
{"ts":"2026-05-19T13:31:38.616Z","level":"info","event":"cron-runner.started","schedule":"0 23 * * *","timezone":"Asia/Shanghai","pid":91513}
{"ts":"2026-05-19T13:32:00.001Z","level":"info","event":"sync.start"}
{"ts":"2026-05-19T13:32:00.450Z","level":"info","event":"sync.done","exitCode":0,"durationMs":449}
{"ts":"...","level":"info","event":"cron-runner.shutdown","signal":"SIGTERM"}
```

JSONL 可直接 `cat | jq` 处理；将来阶段 4 dashboard 可消费此日志做"同步历史"展示。

## 6. 实施过程发现并修复 1 个真实 Bug

### 问题：cron.log 出现重复行

**现象**：每条日志写两次（同 ts、同 pid）。

**初步误判**：以为是 `main()` 被 import 两次（tsx loader 的双初始化）。加了 `globalThis.__DREAM_CRON_BOOTED__` 标志 + 强化 `acquireLock` 同 pid 也抛错 + 让 `main()` silent return —— 但**问题仍然存在**。

**真因**：

`log()` 同时写 `process.stderr` + `appendFileSync(LOG_FILE)`。
而 `start.sh` 用 `nohup ... >> cron.log 2>&1` 启动，`2>&1` 把 stderr 重定向到了 cron.log。
→ 每条 log 既被 stderr 路径写入 cron.log，又被 appendFileSync 路径写入 cron.log，**duplicated**。

**修复**：log 函数只写文件，不写 stderr。前台调试用 `tail -f cron.log`。

**经验**：调试"日志重复"问题，先确认是不是**两条调用路径写到同一个 sink**，而不是想当然以为是"代码被执行两次"。stderr 在 nohup 重定向场景下要谨慎。

> 顺手保留的防御：`acquireLock` 同 pid 也抛错 + `main()` silent return on lock failure。即使将来真发生 tsx 双 init，第二次 main 也不会抢 cron task。
