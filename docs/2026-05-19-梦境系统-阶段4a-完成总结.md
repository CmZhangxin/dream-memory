## 4. 关键技术决策

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| 1 | 数据访问 | better-sqlite3 readonly 直查 | 不依赖 Hermes /search（搜不准且慢）|
| 2 | 中文分词 | @node-rs/jieba | jieba 已被 Hermes 使用，加同款依赖减少环境差异 |
| 3 | 字典加载 | `readFileSync(node_modules/.../dict.txt)` | jieba 包未声明 `exports`，subpath import 失败 |
| 4 | 服务端外置 | `serverExternalPackages` 标记 jieba + better-sqlite3 | 否则 webpack 试图 bundle .node 二进制，失败 |
| 5 | 缓存策略 | 不缓存（`Cache-Control: no-store`）| 数据小（~1500 行），实时聚合 < 200ms |
| 6 | 错误降级 | 每指标独立 try-catch + 整体降级返回 EMPTY_PERSONA_SNAPSHOT | 避免一个失败拖垮全屏 |
| 7 | 视觉风格 | cyan (`#06b6d4`) | 区别于 dream pink / brain purple / mission amber |
| 8 | L1=0 处理 | health.warnings 友好提示，不假装数据 | 透明 |

## 5. 实施过程踩过的坑

### 坑 1：better-sqlite3 ABI 不匹配 Node 22

**现象**：openDbReadonly 返回 null，warning `NODE_MODULE_VERSION 115 vs 127`

**真因**：依赖原本是用 Node 18 编译的 native binding；今天阶段 1 时升级到 Node 22 后没 rebuild

**修复**：`npm rebuild better-sqlite3`

### 坑 2：jieba 字典加载失败

**现象**：CLI 可用，server-side `getTopKeywords()` 返回空数组

**真因**：`@node-rs/jieba` 包没有 `exports` field，无法 ESM `import { dict } from '@node-rs/jieba/dict'`

**修复**：直接 `readFileSync(path.join(process.cwd(), 'node_modules/@node-rs/jieba/dict.txt'))`

### 坑 3：jieba 在 Next.js server bundle 中静默失败

**现象**：next dev 启动后 `/api/persona` 返回 topKeywords=[]，CLI 测脚本同代码工作正常

**真因**：Next.js webpack/turbopack 试图把 `.node` 平台二进制 bundle 到 server chunk → 加载时失败 → catch 静默吞错

**修复**：`next.config.ts` 加 `serverExternalPackages: ["@node-rs/jieba", "better-sqlite3"]`

### 坑 4：readonly 模式下设 pragma 失败

**现象**：openDbReadonly 一启就抛错（早于 ABI mismatch 那次定位）

**真因**：`db.pragma("journal_mode = WAL")` 在 readonly 连接上被拒绝

**修复**：删除该行；写者（Hermes）已经管 WAL，读者无需再设
