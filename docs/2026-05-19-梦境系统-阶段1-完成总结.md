## 实施过程发现的问题与修复

### 问题 1：腾讯 npm 镜像源 tsx@4.22.3 404

**现象**：
```
npm error 404 Not Found - GET https://mirrors.tencent.com/npm/tsx/-/tsx-4.22.3.tgz
```
本机 `npm config get registry` = `https://mirrors.tencent.com/npm/`，腾讯镜像同步不全。

**修复**：
- 即时修复：手工 `npm install --registry=https://registry.npmjs.org/`
- 根本修复：`start.sh` 的 `prepare_memory()` 改为强制 `--registry=https://registry.npmjs.org/`，避免他人 / CI 环境复发
- 文档化：写入 `dream-dashboard/docs/MEMORY-SETUP.md` 「常见问题」章节

### 问题 2：start.sh 总结面板误打印 Memory Gateway URL

**现象**：缺 Key 时降级跳过了 Memory，但末尾总结面板仍打印 `Memory Gateway: http://localhost:8420`。

**修复**：把判断条件从 `[ "$MEMORY_ENABLED" = "true" ]` 改为 `[ "$MEMORY_ENABLED" = "true" ] && curl -s ...health` —— 真启起来才打印。

### 问题 3（已记录，未修复）：stop.sh 误杀非自家进程

**现象**：`stop.sh` 通过 `lsof -ti :3001` 拿所有占用该端口的进程 PID，会把 Chrome 的客户端连接 PID 也算上一起 SIGTERM。Chrome 通常忽略外部 SIGTERM 因此无伤害，但语义上不严谨。

**建议修复**（留给阶段 2 顺手）：用 `lsof -tiTCP:3001 -sTCP:LISTEN` 只杀监听端的进程。

### 问题 4：`npm install --omit=optional` 跳过了平台原生绑定

**现象**：补 Key 后第一次启动 Memory Gateway，超时失败。`gateway.log` 报：

```
Error: Failed to load native binding
  cause: Cannot find module './jieba.darwin-universal.node'
         Cannot find module '@node-rs/jieba-darwin-universal'
```

**根因**：`@node-rs/jieba` 把平台原生绑定（`-darwin-arm64` / `-darwin-x64` / `-linux-*`）按 npm `optionalDependencies` 分发，`--omit=optional` 直接全部跳过 → 加载时找不到 `.node` 文件崩溃。

**修复**：
- start.sh 的 `npm install` 移除 `--omit=optional` 参数（注释里说明原因）
- 手动补装：`cd ~/.dream-memory/tdai-memory-openclaw-plugin && npm install --registry=https://registry.npmjs.org/`
- 修复后 `node_modules/@node-rs/jieba-darwin-arm64/jieba.darwin-arm64.node` 出现，Gateway 正常启动

**经验**：rust/N-API 类原生模块（如 `@node-rs/*`、`@swc/core`、`sharp`、`better-sqlite3`）的平台预编译产物大多走 optional dependencies 分发，**绝不能** `--omit=optional`。

### 问题 5（已记录，未修复）：上游忽略 `MEMORY_TENCENTDB_DATA_DIR` 环境变量

**现象**：start.sh 设置了 `MEMORY_TENCENTDB_DATA_DIR=$HOME/.dream-memory/data`，但 gateway 实际数据目录是 `~/.memory-tencentdb/memory-tdai/`。

**影响**：数据写到了非预期位置，`~/.dream-memory/data/` 是空的。

**留给阶段 2 处理**：阅读上游源码确认正确的环境变量名（可能是 `TDAI_DATA_DIR` 之类），统一目录布局。备份/迁移文档需更新。

### 问题 6（**安全事件 · 用户已知会**）：API Key 在对话中明文出现

**现象**：用户配置 LLM Key 时把真实 sk-xxx 直接发到对话中。

**处理**：
1. 已书面提示用户立即去 https://platform.deepseek.com/api_keys 吊销该 Key 并轮换
2. 阶段1完成总结 / README 不记录 Key 任何片段
3. 后续阶段 2 增加 Skill / 工具时，**所有需要密钥的提示** 必须明确要求用户自己写入 `.env.local`，不要发对话
